SQL Injection Types Explained

Payload: Admin' OR '1'='1

The single quote after "Admin" closes the string. The attacker then adds OR '1'='1, which is a condition that always evaluates to true. The full query becomes WHERE name='Admin' OR '1'='1' — the database returns every row because the OR condition is satisfied.

Result: Full table dump. All secrets exposed.

Payload: '; DROP TABLE secrets; --

; ends the current SQL statement. DROP TABLE secrets is a new statement that deletes the entire table. -- starts a comment, so the rest of the original query (e.g. ')) is ignored.

Result: Data destruction. The table and all its data are removed.

Payload: ' UNION SELECT id, name, data FROM secrets --

UNION combines results from two SELECT statements. The attacker closes the string, adds UNION, and runs a second query. The -- comments out the rest. Both result sets are returned together.

Result: Data from columns or tables the application never intended to expose. Useful when the original query returns few columns.

  • Parameterized queries — Use placeholders (?, %s) and bind values separately. The database driver escapes input. Never concatenate user input into SQL.
  • Input validation — Whitelist allowed characters and formats. Reject unexpected input early.
  • Least privilege — Database user should have minimal permissions. Avoid DROP, ALTER, or admin rights for app connections.
  • ORM — Use an ORM (SQLAlchemy, Django ORM) that handles escaping automatically.
  • WAF & monitoring — Web application firewalls and logging can detect and block common injection patterns.
About This Demo
Vulnerable Endpoint

This endpoint is vulnerable to SQL injection because it directly concatenates user input into the SQL query. Attackers can manipulate the input to execute arbitrary SQL commands.

# Vulnerable Code Example
query = f"SELECT * FROM secrets WHERE name = '{user_input}'"
cursor.execute(query)

SQL injection remains one of the most critical web vulnerabilities. OWASP ranks it in the top 3.

Consequences: Data breaches (millions of records), account takeover, credential theft, data destruction, and full database compromise. Real attacks have affected major companies and exposed sensitive user data.

Secure Endpoint

This endpoint is protected against SQL injection by using parameterized queries, which properly escape and sanitize user input before including it in SQL statements.

# Secure Code Example
query = "SELECT * FROM secrets WHERE name = ?"
cursor.execute(query, (user_input,))
  • Parameterized queries — Never concatenate user input into SQL
  • Input validation — Whitelist allowed characters and formats
  • Least privilege — DB user should have minimal permissions
  • ORM — Use an ORM that handles escaping automatically
